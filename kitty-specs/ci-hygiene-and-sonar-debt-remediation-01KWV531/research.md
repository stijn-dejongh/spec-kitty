# Phase 0 Research: CI Hygiene & Sonar Debt Remediation

## R1 — `_baselines.yaml` schema for the census LOC-ratchet migration (IC-01)

**Decision**: Add a new top-level key to `tests/architectural/_baselines.yaml` (e.g. `test_ci_topology_worklist`) with one integer sub-key per tracked worklist directory's committed LOC baseline (e.g. `session_presence_loc: 1229`). `test_ci_topology_worklist.py`'s assertion changes from `census["worklist"] == gc.live_derived_worklist()` (exact equality over the *whole* structure, including LOC) to two independent checks: (a) the *structural* fields (`dir`, `cone_roots`, `target_group`, `target_shard`) still compared by exact equality against live derivation — these only change when routing topology genuinely changes, which is exactly what NFR-001 requires to keep catching; (b) each dir's live LOC compared against its `_baselines.yaml` entry via the existing ratchet meta-test (`test_ratchet_baselines.py`) — growth fails, shrinkage warns, exactly like the other 7 gated modules.

**Rationale**: Confirmed live (`kitty-specs/slice-f-multi-context-extensibility-01KRX5C8/contracts/ratchet-baseline-format.md`) that `_baselines.yaml`'s schema is `{test_module_name: {baseline_key: int}}` — a flat per-key integer count, validated by a `BaselinesFile` pydantic model and a committed contract doc. This is a direct structural fit for "one LOC baseline per tracked directory" — no schema extension needed, just a new top-level entry following the existing convention. The `test_every_worklist_dir_meets_loc_floor` test already referenced in the spec's Edge Cases treats LOC as a floor check for shard-assignment purposes (a dir must be *big enough* to justify its own routing entry) — that check is orthogonal to this ratchet (a floor is "big enough to matter", a ratchet is "hasn't silently grown unnoticed") and both can coexist unchanged.

**Alternatives considered**:
- *Tolerance band on the census file itself* (e.g. `abs(committed_loc - live_loc) < N`) — rejected: introduces a new brittle magic-number constant (what's N?) that itself needs periodic re-tuning, exactly the failure mode the spec's Edge Cases warn against.
- *Drop LOC tracking from CI-topology entirely* — rejected: LOC is load-bearing for shard/group assignment elsewhere in the same census (`target_shard`/`target_group` decisions may depend on directory size crossing `t_loc`); removing it outright would require redesigning the routing-assignment logic itself, out of scope for this mission (see C-001-adjacent scope discipline — don't drive-by-refactor the census's routing logic, only its brittleness).

## R2 — Canonical contract-path-resolution helper placement (IC-02)

**Decision**: New shared helper as a `pytest` fixture in a `conftest.py` at the lowest common ancestor of the two consuming test files — `tests/specify_cli/conftest.py` (parent of both `tests/specify_cli/cli/commands/` and `tests/specify_cli/compat/`). Resolution logic: walk up from the fixture's own `__file__` to the actual repository root by finding the nearest ancestor directory containing `pyproject.toml` (a stable, environment-agnostic marker), then descend into `kitty-specs/cli-upgrade-nag-lazy-project-migrations-01KQ6YDN/contracts/compat-planner.json` from there. Raise (not silently return `None`) if the contract file still isn't found after that resolution, satisfying FR-004's "fail loudly" requirement.

**Rationale**: A `pyproject.toml`-anchored walk is checkout-shape-agnostic by construction — it doesn't care whether the test runs from a plain clone, a CI runner's nested work directory, or a maintainer's `.worktrees/<name>/` layout, because it climbs until it finds the actual marker file rather than assuming a fixed number of parent hops. This directly fixes both files' root cause (each hardcodes a *specific hop count* that happens to work in exactly one layout). Placing it in `conftest.py` as a fixture (not a bare module-level constant) also means pytest's collection makes it available to both files without either needing to import from the other, and follows this repo's existing pytest-fixture-sharing convention.

**Alternatives considered**:
- *A plain importable function in a new `tests/_contract_paths.py` module* — viable and simpler for non-fixture use, but the two consuming files use the path in different shapes (`test_upgrade_command.py` needs a module-level constant computed at import time for its `_CONTRACT_PATH.exists()` guard pattern; `test_messages.py` needs it inside a helper method). A `conftest.py` fixture accommodates both via `request.getfixturevalue` or a session-scoped fixture imported once, without forcing either file to restructure its existing usage pattern more than necessary.
- *Environment variable / pytest CLI flag* — rejected: adds operational complexity (something to remember to set) for a problem that has a purely structural, environment-agnostic solution.

## R3 — Sonar/internal coverage-scope reconciliation (IC-05)

**Decision**: Treat this as research-first, not fix-first. First task: attempt to determine, using only the unauthenticated public SonarCloud API (confirmed accessible during the post-spec validation squad — no `SONAR_TOKEN` needed for this public project's read endpoints), whether SonarCloud's `coverage` metric is measuring a materially different file set than the internal `diff-coverage` gate (e.g., via `GET /api/measures/component_tree` scoped to `coverage` and comparing the component list against the internal gate's critical-path file list) — not just a different *threshold* on the same files. If the file sets differ, document the difference (which files/dirs Sonar includes that the internal gate excludes, or vice versa) as the primary FR-007 deliverable. If the file sets are actually the same and only the *threshold philosophy* differs (whole-repo average vs. per-PR-diff-only), the fix is purely explanatory documentation, not a scope-alignment code change.

**Rationale**: The spec's own User Story 5 already concedes this mission "cannot verify without file-level Sonar API access" whether a genuine instrumentation gap exists — but the squad's live verification found that access is available unauthenticated for this public repo, which the original spec author didn't know when filing #2422. This research task closes that gap before committing to either a "documentation-only" or "code-fix" path.

**Alternatives considered**: None yet meaningful to compare — this IC is explicitly a research-then-decide task, not a predetermined design choice.

## R4 — `sonarcloud_branch_review.sh` promotion target and idempotency design (IC-06)

**Decision**: Promote to `scripts/ci/sonarcloud_branch_review.sh` (this repo's existing convention for CI-adjacent scripts — sibling to `scripts/ci/quality_gate_decision.py`, confirmed present during the post-spec squad's Area 4 investigation). Idempotency for the backlog-slicing use (IC-07) is layered on *top* of the promoted script (which itself stays a read-only query tool), via a companion filing step that checks each candidate issue-slice against a `gh issue list --search "<rule-id> <file-path> in:body"`-style query (or a locally-cached manifest of already-filed slices, whichever proves more reliable during implementation) before creating a new issue — skip-if-already-filed per the confirmed planning decision.

**Rationale**: Keeping the promoted script itself a pure, stateless SonarCloud-query tool (its current shape) and layering idempotency in the *filing* step (which is inherently mission-specific: "have I already ticketed this slice?") keeps the two concerns — "talk to SonarCloud" vs. "don't duplicate GitHub issues" — cleanly separated, consistent with DIRECTIVE_044's unification principle (one tool, one job) rather than building a stateful, GitHub-aware SonarCloud client.

**Alternatives considered**:
- *Rewrite as a Python module instead of promoting the existing shell script* — rejected per paula-patterns' explicit finding during spec validation: "resisting the urge to prescribe re-architecture... the fold-in is 'use the existing script,' not 're-architect it.'" A rewrite is unjustified scope expansion for a script that already works.
- *Idempotency baked into the SonarCloud-query script itself* (e.g., it tracks its own "already seen" state) — rejected: conflates a stateless query tool with a stateful filing tracker, harder to test in isolation, and couples the promoted tool to GitHub's API for something SonarCloud's API has no opinion about.

## R5 — Epic parenting mechanics for FR-008 backlog-slice issues

**Decision**: File each backlog-slice issue directly with `gh issue create`, then attach it as a native GitHub sub-issue of #1928 via `gh api repos/Priivacy-ai/spec-kitty/issues/1928/sub_issues -X POST -F sub_issue_id=<new-issue-node-id>` — the same mechanism already used successfully multiple times this session (e.g., parenting #2416/#2419 under #1931, #2421/#2422 under #825). `sub_issue_id` must be the issue's numeric `id` field (from `gh api repos/.../issues/<n> --jq '.id'`), not its display number — confirmed via a prior same-session mistake (a wrong numeric ID was caught and corrected during this session's #2412/#1619 parenting).

**Rationale**: Reuses an already-proven, already-debugged mechanism from earlier in this same session rather than re-deriving the GitHub sub-issues API shape from scratch.

**Alternatives considered**: Checklist-reference-only parenting (a line in #1928's body linking to each new issue) — rejected as the primary mechanism per this session's established convention of preferring native sub-issues where GitHub's single-parent constraint allows it; checklist-reference remains the fallback only if a given issue already has a different native parent (per GitHub's single-parent-per-issue constraint), matching how #2183/#2185 etc. were handled earlier in the coord-authority epic.
