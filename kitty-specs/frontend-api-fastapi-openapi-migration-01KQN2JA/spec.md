**Feature Slug**: `frontend-api-fastapi-openapi-migration-01KQN2JA`

**Mission Type**: software-dev
**Created**: 2026-05-02
**Target Branch**: `feature/650-dashboard-ui-ux-overhaul`

## Overview

Steps 3 and 4 of the Frontend Decoupling and Application API Platform epic
([#645](https://github.com/Priivacy-ai/spec-kitty/issues/645)) — contract
formalisation/hardening and FastAPI/OpenAPI transport migration
([#460](https://github.com/Priivacy-ai/spec-kitty/issues/460)) — combined
into a single mission so the same response-model substrate (Pydantic) feeds
both the typed-Python contracts and the auto-generated OpenAPI document.

This mission is unblocked because:

- **Step 2 (handler-to-service extraction)** completed in mission
  `dashboard-service-extraction-01KQMCA6` (#111). All UI-facing read/command
  logic now lives in `src/dashboard/services/*.py` and `src/dashboard/file_reader.py`.
  The handler layer in `src/specify_cli/dashboard/handlers/` is a thin
  single-call adapter (post follow-up `dashboard-extraction-followup-01KQMNTW`).
- **Charter bundle chokepoint** — the second hold reason from #460 — completed
  in mission `unified-charter-bundle-chokepoint-01KP5Q2G`. `ensure_charter_bundle_fresh()`
  is wired through every charter-derivative reader.

The mission ships a FastAPI-backed transport mounted on the existing
`spec-kitty dashboard` CLI surface, an auto-generated OpenAPI document
served alongside it, and Pydantic response models that retain shape parity
with the existing `TypedDict` contracts. The strangler-fig pattern preserves
behavior at every step: the legacy `BaseHTTPServer` + hand-rolled router can
coexist with the FastAPI app during the transition, with a feature flag /
config switch flipped to FastAPI once parity is proven.

The mission also lays the groundwork for downstream MCP exposure of these
routes: route handler functions are written so they can be invoked as plain
Python callables (no HTTP-only side effects in the handler body) so a future
MCP adapter can re-use them without re-implementing the contract.

## Domain Language

| Term | Canonical meaning |
|------|-------------------|
| **FastAPI app** | The new transport layer (`fastapi.FastAPI` instance) that replaces `DashboardHTTPServer` + `DashboardRouter`. |
| **OpenAPI document** | The OpenAPI 3.x JSON/YAML auto-generated by FastAPI from route signatures and Pydantic models. Served at `/openapi.json` and `/docs` (Swagger UI) on the dashboard surface. |
| **Response model** | A Pydantic `BaseModel` class that defines the JSON shape FastAPI returns for a given route. Replaces the corresponding `TypedDict` once parity is proven. |
| **Contract parity** | The condition where every existing dashboard route's JSON output is byte-equivalent (modulo key ordering) before and after migration. Verified by a parity test suite. |
| **Strangler boundary** | The conditional dispatch point that decides whether a request is served by the legacy `BaseHTTPServer` stack or the FastAPI app. Lives in `src/specify_cli/dashboard/server.py` during migration. |
| **MCP-friendly handler** | A FastAPI route function whose body has no HTTP-only side effects (no `Response` writes, no header mutations) so it can be invoked as a plain Python callable. |

## User Scenarios & Testing

### User Story 1 — Frontend / SDK consumer reads the OpenAPI spec (Priority: P0)

A developer (internal or external) wants to build a UI or CLI client that
reads dashboard data. They request `GET /openapi.json` from a running
`spec-kitty dashboard` instance and receive a complete, valid OpenAPI 3.x
document describing every dashboard route, query parameter, response shape,
and status code. They feed the spec into `openapi-typescript` (or any
similar codegen tool) and receive typed bindings that compile cleanly.

### User Story 2 — Operator runs the dashboard with no observable change (Priority: P0)

An operator runs `spec-kitty dashboard` exactly as before. They see the same
URLs, the same JSON response shapes, the same browser UI. They do not need
to know FastAPI is now serving the requests. The startup time, memory
footprint, and behavior under all existing dashboard tests are unchanged
within the NFR thresholds.

### User Story 3 — Future MCP integrator re-uses route handlers (Priority: P1)

A future contributor wants to surface dashboard routes through MCP. They
inspect the FastAPI route handler functions and discover that each one is a
plain Python function whose return value is a Pydantic model — no
`fastapi.Response` writes, no header mutation. They can wrap each route
function in an MCP tool definition that calls the handler directly and
returns the Pydantic model serialised as JSON, with no HTTP layer involved.

### User Story 4 — Reviewer audits contract drift (Priority: P1)

A reviewer wants to confirm that no JSON response shape changed during the
transport migration. They open the parity test suite and see one
parametrized test per route that asserts the old and new responses match.
A separate test compares the generated OpenAPI document against a golden
snapshot to catch unintended schema drift.

### Edge cases

- **Token validation on `/api/sync/trigger`** — token validation must remain
  in a place where it can run before the route function body. FastAPI
  dependencies (`Depends`) are the canonical place for this. The dependency
  result feeds into the route function as a typed argument, so the
  MCP-friendly invariant (no HTTP-only side effects in handler body) holds
  even though the dependency itself is HTTP-aware.
- **Static asset serving (`handlers/static.py`)** — FastAPI's
  `StaticFiles` mount replaces the existing static handler. Asset paths
  remain unchanged from the operator's perspective.
- **WebSocket / SSE for live updates** — out of scope for this mission;
  Step 5 of the epic. Note: FastAPI supports these natively, so this
  mission's transport choice keeps Step 5 cheap.
- **Existing legacy handlers (`handlers/glossary.py`, `handlers/lint.py`,
  `handlers/charter`, `handlers/dossier`)** — these must migrate too. Some
  may not yet be fully service-extracted (the prior mission targeted
  features/api/health/sync). Routes that retain handler-layer logic move
  into FastAPI as-is for transport parity, with a follow-up backlog item to
  service-extract them in the next mission.
- **OpenAPI document determinism** — the same code on the same CPython
  produces a byte-equivalent OpenAPI document; the parity test relies on
  this for the golden-snapshot check.

## Key Entities

- **`FastAPI` app instance** — `src/dashboard/api/app.py` exports `create_app(project_dir, project_token) -> FastAPI`. Constructor injects the same dependencies the legacy handler layer pulled out of class attributes.
- **Pydantic response models** — `src/dashboard/api/models.py`. Each TypedDict in `src/dashboard/api_types.py` gets an equivalent `BaseModel`. Models reuse the canonical name from `api_types.py` (with the file split sufficient to disambiguate).
- **Route modules** — `src/dashboard/api/routers/{features,kanban,artifacts,health,sync,charter,diagnostics,dossier,glossary,lint,shutdown}.py`. Each module exports a `fastapi.APIRouter` registered on the app.
- **Strangler boundary** — `src/specify_cli/dashboard/server.py`. Reads `dashboard.transport` config; selects which stack to run. CLI flags `--transport` override the config.
- **OpenAPI snapshot** — `tests/test_dashboard/snapshots/openapi.json` (golden, committed) versus the runtime-generated document.
- **Parity test fixture** — runs the legacy stack and the FastAPI app side-by-side against the same fixture project; asserts byte-equivalent JSON per route.
- **Migration runbook** — `docs/migration/dashboard-fastapi-transport.md`. Operator-facing doc.

## Assumptions

- **Pydantic v2** is the target version; if Pydantic v1 is still pinned anywhere in `pyproject.toml`, the upgrade is in scope (additive, not breaking — Pydantic v2 is widely adopted by FastAPI's current release line).
- **Uvicorn** is acceptable as the ASGI server; the alternative (Hypercorn) is not chosen unless a concrete reason arises during planning. Uvicorn is the FastAPI team's default recommendation.
- **Handler-extraction completeness**: The `src/dashboard/services/` package covers the four primary read/command surfaces (mission scan, project state, sync, file reader). Glossary, lint, charter, dossier handlers may not yet have equivalent service objects. For this mission, those routes migrate transport-only (the handler-layer logic comes along for the ride). Service-extracting them is explicitly out of scope and a separate mission.
- **The dashboard remains local-first**: no auth scheme change, no exposure to non-loopback network interfaces. The strangler boundary preserves the existing localhost binding.
- **No existing OpenAPI document**: the dashboard does not currently publish OpenAPI; this mission is greenfield for that artifact.
- **Existing `tests/test_dashboard/test_api_contract.py`** stays in place. The TypedDict-based contract test continues to pass for any consumer still importing TypedDicts. The new parity + snapshot tests sit alongside it, not replace it.

## Governance

### Active Directives

- **DIRECTIVE_024** (Locality of Change) — allowed scope: `src/dashboard/api/` (new code), `src/dashboard/api_types.py` (TypedDict-side compat), `src/specify_cli/dashboard/server.py` (strangler boundary + CLI flag wiring), `tests/test_dashboard/`, `tests/architectural/`, `architecture/2.x/`, `docs/migration/`, `kitty-specs/<this-mission>/`.
- **DIRECTIVE_010** (Test Coverage Discipline) — every new route function and every Pydantic model must have at least one test that exercises a non-trivial path; the parity suite + OpenAPI snapshot test together meet this for transport.
- **DIRECTIVE_036** (Adapter Test Pattern) — seam tests for the new FastAPI handlers must use FastAPI's `TestClient` rather than asserting against `urllib`-level state; this is consistent with the existing seam-test pattern in the parent extraction mission.

### Applied Tactics

- `strangler-fig-extraction` — the strangler boundary in `server.py` and the dual-stack co-existence period.
- `golden-snapshot-testing` — for the OpenAPI document.
- `transport-parity-suite` — for verifying response equivalence during migration.
- `adr-drafting-workflow` (via `how-we-apply-directive-003`) — for the FastAPI/OpenAPI ADR (FR-015).

## Out of Scope (explicit non-goals)

The following are intentionally not addressed by this mission:

- **NG-1**: No frontend (`dashboard.js`) refactor beyond updating any URL paths or response-shape access patterns that change due to migration. The monolithic `dashboard.js` controller stays as-is.
- **NG-2**: No new product-surface endpoints. Only existing routes migrate.
- **NG-3**: No WebSocket / SSE async update transport (Step 5 of the epic).
- **NG-4**: No service extraction for handlers that still hold inline logic (glossary, lint, charter, dossier) — separate mission.
- **NG-5**: No public exposure of the API beyond localhost. The dashboard remains a local development tool.
- **NG-6**: No MCP server implementation. The mission lays the groundwork (FR-009, FR-020) but does not ship an MCP adapter.
- **NG-7**: No retirement of the legacy `BaseHTTPServer` stack. That is a separate, sequenced follow-up mission.
- **NG-8**: No auth/authz redesign. The token-validation pattern stays the same shape (a Depends dependency replaces the inline guard).
