# Implementation Plan: Dashboard Service Extraction

*Path: [kitty-specs/dashboard-service-extraction-01KQMCA6/plan.md](plan.md)*

**Branch**: `feature/650-dashboard-ui-ux-overhaul` | **Date**: 2026-05-02
**Spec**: [spec.md](spec.md) | **Mission**: `dashboard-service-extraction-01KQMCA6`

## Summary

Extract all dashboard query and orchestration logic from the five multiple-inheritance HTTP
handler classes into a new canonical `src/dashboard/` package, following the strangler-fig
extraction pattern established by `src/charter/` and `src/doctrine/`. The HTTP handler layer
in `src/specify_cli/dashboard/handlers/` becomes a thin delegation adapter. `api_types.py`
relocates to `src/dashboard/` with a backward-compat shim at the old path. An architectural
boundary test asserts no circular imports between the service layer and its adapter.
`dashboard.js` is adapted to consume the stabilised endpoint contracts without behavioral
change.

## Technical Context

**Language/Version**: Python 3.11+
**Primary Dependencies**: stdlib only (`pathlib`, `threading`, `json`, `http.server`); no new
PyPI packages. Service layer reads from existing `specify_cli.scanner`, `specify_cli.status`,
`specify_cli.diagnostics`, `specify_cli.sync.daemon`, `specify_cli.glossary`,
`specify_cli.dossier.api`.
**Storage**: Filesystem only — `kitty-specs/` mission artifacts, `.kittify/` config,
`.worktrees/` git worktrees, charter/doctrine YAML files.
**Testing**: pytest; fixture-based approach for filesystem and scanner dependencies;
`pytestarch` for architectural layer assertions (existing pattern in `tests/architectural/`).
**Target Platform**: Linux server (local operator tool running in the same environment as the
CLI).
**Project Type**: CLI library — `src/` layout, no build step, editable install via `uv`.
**Performance Goals**: Dashboard startup ≤5% regression (NFR-001); test suite wall-clock
≤10% increase after new seam and boundary tests (NFR-004).
**Constraints**: BaseHTTPServer transport retained; `api_types.py` shapes unchanged; no new
PyPI dependencies; `spec-kitty dashboard` CLI surface unchanged; `dashboard.js` behavior
unchanged from operator perspective.
**Scale/Scope**: 15 HTTP routes across 5 handler classes (≈780 lines of handler code);
440 lines of TypedDict contracts in `api_types.py`; 16 existing tests in `tests/test_dashboard/`.

## Engineering Approach

### Service Boundary Design

The handler code contains three responsibility clusters above the pure-I/O threshold
(connascence analysis in `research.md`):

| Service Object | Responsibility | Source methods |
|---|---|---|
| `MissionScanService` | Scans all missions, resolves active mission, assembles kanban state with weighted progress | `handle_features_list`, `handle_kanban` |
| `ProjectStateService` | Assembles project health: path resolution, sync daemon status query | `handle_health` |
| `SyncService` | Orchestrates sync daemon startup and trigger | `handle_sync_trigger` |

Methods already below the extraction threshold (thin I/O delegation):
`handle_diagnostics`, `handle_charter`, `handle_dossier`, all file-serving routes
(`handle_research`, `handle_artifact`, `handle_contracts`, `handle_checklists`).
These remain in the adapter layer per C-006.

### `api_types.py` Relocation

`api_types.py` moves from `src/specify_cli/dashboard/api_types.py` to
`src/dashboard/api_types.py`. The old path becomes a shim:

```python
# src/specify_cli/dashboard/api_types.py (shim)
from dashboard.api_types import *  # noqa: F401, F403  # removal_release: FastAPI milestone
```

All existing import paths in `src/specify_cli/` resolve through the shim transparently.
No shape or field name changes (C-002).

### Architectural Boundary Test Scope (FR-010 calibration)

FR-010 targets the practical boundary for this mission:

> No module inside `src/dashboard/` imports from `src/specify_cli/dashboard/`

The broader "no `specify_cli` imports" constraint is achievable only after scanner and status
subsystems are themselves extracted to top-level packages (future missions). The ADR
documents this staged approach. `test_dashboard_boundary.py` asserts the narrower rule.
`test_layer_rules.py` is updated to register `dashboard` as a known layer.

### Extraction Sequence (strangler-fig, per-route)

| WP | Work | Dependencies |
|---|---|---|
| WP01 | Ownership map entry + ADR | None |
| WP02 | Scaffold `src/dashboard/`; relocate `api_types.py`; register layer in architectural test | WP01 |
| WP03 | Extract `MissionScanService` + `DashboardFileReader`; thin `features.py` handlers | WP02 |
| WP04 | Extract `ProjectStateService` + `SyncService`; thin `handle_health` + `handle_sync_trigger` in `api.py` (merged: both touch `api.py`, parallel would conflict) | WP02 |
| WP05 | Add `test_dashboard_boundary.py` + `test_seams.py`; verify all existing tests pass | WP03, WP04 |
| WP06 | `dashboard.js` field audit + smoke-test running dashboard | WP05 |

WP03 and WP04 are parallel-eligible (WP03 owns `features.py`; WP04 owns `api.py`).

### Router Structure (preserved)

`DashboardRouter`'s multiple-inheritance chain is not changed. Each handler class is
individually thinned; routing topology stays identical. This keeps `test_router_dispatch.py`
unmodified and minimises blast radius.
