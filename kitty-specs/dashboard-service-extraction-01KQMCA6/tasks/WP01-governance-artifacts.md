2. Write the ADR with the following sections:

   ```markdown
   # ADR: Dashboard Service Extraction

   **Date**: 2026-05-02
   **Status**: Accepted
   **Mission**: `dashboard-service-extraction-01KQMCA6`

   ## Context

   The Spec Kitty dashboard handler layer (`src/specify_cli/dashboard/handlers/`)
   currently embeds all business logic â€” mission scanning, kanban assembly,
   health status, sync orchestration â€” directly inside HTTP request handlers.
   This coupling makes the logic untestable in isolation and impossible to
   migrate to a modern transport framework (FastAPI/OpenAPI, tracked as a
   sequenced follow-up mission).

   The connascence analysis in `kitty-specs/dashboard-service-extraction-01KQMCA6/research.md`
   identifies three high-strength dynamic connascence clusters in the handler
   layer that exceed the extraction threshold:
   - `handle_features_list` / `handle_kanban` â€” execution-order coupling across
     scanner, legacy detector, and status subsystems
   - `handle_sync_trigger` â€” three-step orchestration mixing HTTP response
     concerns with daemon state
   - `handle_health` â€” status dict shape depends on daemon response shape,
     coupled in the same method

   The dashboard is also absent from the functional ownership map, creating a
   governance gap.

   ## Decision

   Extract dashboard business logic into a new canonical top-level package
   `src/dashboard/`, following the strangler-fig extraction pattern established
   by `src/charter/` and `src/doctrine/`. Three service objects are introduced:

   - `MissionScanService` â€” features list, kanban with weighted progress
   - `ProjectStateService` â€” health response assembly
   - `SyncService` â€” sync daemon orchestration

   `api_types.py` moves to `src/dashboard/api_types.py`. A backward-compat
   shim is installed at the old path with `removal_release: FastAPI milestone`.

   The handler layer in `src/specify_cli/dashboard/handlers/` becomes a thin
   delegation adapter. Each handler method contains only HTTP dispatch and a
   single call to a service object.

   ## Rationale

   - Resolves high-strength dynamic connascence (execution-order coupling) by
     giving each cluster a stable, testable interface
   - Enables isolated unit testing of dashboard logic without an HTTP server
   - Unblocks the FastAPI transport migration (sequenced follow-up mission)
   - Follows the established extraction pattern from charter and doctrine
   - Registers the dashboard as a first-class governance slice

   ## Consequences

   - `src/dashboard/` imports from `specify_cli.scanner`, `specify_cli.status`,
     `specify_cli.sync.daemon` â€” these are intentional cross-layer dependencies
     on subsystems not yet extracted; full boundary isolation requires scanner
     and status extraction missions
   - The architectural boundary test (`test_dashboard_boundary.py`) asserts
     the narrower invariant: no imports from `src/specify_cli/dashboard/`
     inside `src/dashboard/` (prevents circular dependency)
   - `DashboardRouter`'s multiple-inheritance chain is unchanged

   ## Rejected Alternatives

   - **One monolithic `DashboardService`**: rejected because `MissionScanService`
     and `SyncService` have entirely different dependencies; bundling them
     inflates constructor coupling (see `research.md Â§2`)
   - **One service object per handler class**: rejected because `FeatureHandler`
     has two responsibility clusters (scan and file I/O); the file-serving
     cluster does not belong in a service object
   - **Keep `api_types.py` in `src/specify_cli/dashboard/`**: rejected because
     it would force service objects to import from `specify_cli`, creating a
     circular dependency
   ```
